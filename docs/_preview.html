Multi-Provider Agent System

Design document for supporting multiple LLM providers (Claude, OpenAI, Grok/xAI) with custom personalities per group.

Status: Proposed
Date: 2026-02-22

Problem Statement

CamBot-Agent currently runs every agent on Claude via the Anthropic SDK. This works well for a single-personality assistant, but limits several use cases:

  1. Model selection per group — Some groups may benefit from a different model's strengths (GPT for creative writing, Grok for real-time web knowledge, Claude for code).
  2. Agent councils — Multiple agents with different providers deliberate on a problem, each bringing a different perspective.
  3. Specialized agents — A code reviewer (Claude), a creative writer (GPT), and a researcher (Grok) each handling what they're best at.
  4. Cost optimization — Route lightweight tasks to cheaper models, reserve expensive models for complex work.
  5. Redundancy — If one provider has an outage, groups can fall back to another.

The goal is to support this without rewriting the host orchestration layer, which is already provider-agnostic.

The Seam

The container boundary (stdin JSON in, stdout JSON out) is the natural abstraction point. The host already treats containers as black boxes. The only host-side coupling is:

  1. readSecrets() — which secrets to inject
  2. CONTAINER_IMAGE — which image to spawn
  3. buildVolumeMounts() — .claude/settings.json creation (Claude-specific)

Proposed Design

Agent Definition as a First-Class Concept

Introduce an AgentDefinition that describes an agent's identity independent of the group it serves:

interface AgentDefinition {
  id: string;               // e.g., "claude-default", "gpt-creative", "grok-researcher"
  provider: 'claude' | 'openai' | 'grok';
  model: string;            // e.g., "claude-sonnet-4-6", "gpt-4o", "grok-3"
  personality?: string;     // System prompt overlay (appended to base prompt)
  containerImage: string;   // e.g., "cambot-agent-claude:latest", "cambot-agent-openai:latest"
  secretKeys: string[];     // Which env vars to inject (e.g., ["OPENAI_API_KEY"])
}

Groups reference agents by ID:

interface RegisteredGroup {
  // ... existing fields ...
  agentId?: string;  // Default: "claude-default" (backwards compatible)
}

One Container Image Per Provider (Open-Closed Principle)

Each provider gets its own container image with its own agent-runner implementation. Adding a new provider means building a new image — not modifying existing ones.

container/
  agent-runner-claude/     # Existing agent-runner (renamed)
    src/index.ts           # Uses @anthropic-ai/claude-agent-sdk
    src/ipc-mcp-stdio.ts   # Shared (symlinked or copied)
    Dockerfile
  agent-runner-openai/     # New
    src/index.ts           # Uses openai SDK
    src/ipc-mcp-stdio.ts   # Same MCP server
    Dockerfile
  agent-runner-grok/       # New (thin wrapper — Grok uses OpenAI-compatible API)
    src/index.ts           # Uses openai SDK with xAI base URL
    src/ipc-mcp-stdio.ts   # Same MCP server
    Dockerfile

The Container Protocol (Abstraction Boundary)

The existing stdin/stdout JSON protocol is the contract. Every provider container must implement:

Input (stdin, single JSON blob):

{
  "prompt": "User's message",
  "sessionId": "optional-resume-id",
  "groupFolder": "family-chat",
  "chatJid": "120363...@g.us",
  "isMain": false,
  "isScheduledTask": false,
  "secrets": { "OPENAI_API_KEY": "sk-..." }
}

Output (stdout, wrapped in sentinel markers):

---CAMBOT_AGENT_OUTPUT_START---
{"status":"success","result":"Agent's response","newSessionId":"sess-abc"}
---CAMBOT_AGENT_OUTPUT_END---

IPC (file-based, unchanged):

  • /workspace/ipc/messages/ — outbound messages
  • /workspace/ipc/tasks/ — task management
  • /workspace/ipc/input/ — inbound follow-up messages, _close sentinel

This protocol is already clean. No changes needed.

Host-Side Changes Summary

1. Types (src/types.ts)

Add AgentDefinition interface. Add agentId?: string to RegisteredGroup.

2. Database (src/db.ts)

New table:

CREATE TABLE IF NOT EXISTS agent_definitions (
  id TEXT PRIMARY KEY,
  provider TEXT NOT NULL,
  model TEXT NOT NULL,
  personality TEXT,
  container_image TEXT NOT NULL,
  secret_keys TEXT NOT NULL  -- JSON array
);

Add agent_id column to registered_groups (default: 'claude-default').

Composite session key — add agent_id to the sessions table:

-- Migration: sessions table gets composite key
CREATE TABLE sessions_v2 (
  group_folder TEXT NOT NULL,
  agent_id TEXT NOT NULL DEFAULT 'claude-default',
  session_id TEXT NOT NULL,
  PRIMARY KEY (group_folder, agent_id)
);

3. Config (src/config.ts)

Remove CONTAINER_IMAGE global constant. Image is now per-agent from the definition.

4. Container Runner (src/container-runner.ts)

  • readSecrets() becomes readSecrets(secretKeys: string[]) — reads only the keys the agent needs.
  • buildContainerArgs() takes containerImage: string parameter instead of using the global.
  • buildVolumeMounts() — skip .claude/settings.json for non-Claude providers. Each provider's Dockerfile handles its own config.
  • runContainerAgent() signature gains agentDef: AgentDefinition parameter.

5. Orchestrator (src/index.ts)

When processing messages for a group:

  1. Look up group.agentId (default 'claude-default').
  2. Fetch AgentDefinition from DB.
  3. Pass definition to runContainerAgent().

6. Secrets (.env)

Add provider-specific keys alongside existing ones:

ANTHROPIC_API_KEY=sk-ant-...
OPENAI_API_KEY=sk-...
XAI_API_KEY=xai-...

Each agent definition declares which keys it needs via secretKeys.

Session Management Across Providers

Sessions are currently keyed by group_folder alone. With multi-provider support, the key becomes (group_folder, agent_id):

  • Switching a group from Claude to GPT starts a fresh session (different agent, different session).
  • A council scenario runs multiple agents in the same group — each maintains its own session.
  • The existing getSession()/setSession() API gains an agentId parameter.

Provider Containers

Claude (Existing — Minimal Changes)

Rename agent-runner/ to agent-runner-claude/. The existing code is nearly unchanged:

  • index.ts — already correct. Uses @anthropic-ai/claude-agent-sdk.
  • SECRET_ENV_VARS — stays ['ANTHROPIC_API_KEY', 'CLAUDE_CODE_OAUTH_TOKEN'].
  • Session resume — uses SDK's resume/resumeSessionAt (Claude-specific).
  • Hooks — PreCompact and PreToolUse hooks remain.
  • Tools — full Claude Code toolset (Bash, Read, Write, Edit, etc.).

The .claude/settings.json setup moves into this image's entrypoint (out of host container-runner.ts).

OpenAI (New Image, ~200 Lines)

New agent-runner-openai/src/index.ts:

// Pseudocode — same stdin/stdout protocol, different SDK
import OpenAI from 'openai';

const client = new OpenAI({ apiKey: secrets.OPENAI_API_KEY });

// Read ContainerInput from stdin (identical to Claude runner)
// Start MCP server (identical — same ipc-mcp-stdio.ts)
// Convert MCP tools to OpenAI function-calling format
// Run completion loop with tool calls
// Write ContainerOutput to stdout with sentinel markers

Key differences from the Claude runner:

  • No session resume (OpenAI has no equivalent — session context is rebuilt from conversation history stored in the group folder).
  • Tool calling uses OpenAI's function-calling API instead of Claude's tool_use blocks.
  • System prompt is passed directly (no systemPrompt.preset).
  • No hooks system — bash sanitization is done inline.

The MCP server (ipc-mcp-stdio.ts) is shared unchanged — it's pure file-based IPC.

Grok/xAI (OpenAI-Compatible, Trivial Delta)

Grok's API is OpenAI-compatible. The Grok runner is a thin wrapper:

import OpenAI from 'openai';

const client = new OpenAI({
  apiKey: secrets.XAI_API_KEY,
  baseURL: 'https://api.x.ai/v1',
});

// Everything else identical to OpenAI runner

Could even be the same image with a BASE_URL environment variable, but separate images are cleaner (open-closed).

Shared MCP Server

ipc-mcp-stdio.ts is already provider-agnostic. It:

  • Reads context from environment variables (CAMBOT_AGENT_CHAT_JID, etc.)
  • Writes JSON files to /workspace/ipc/
  • Has zero SDK imports

Each provider runner spawns it identically. For providers without native MCP support (OpenAI, Grok), the runner converts MCP tool definitions to the provider's function-calling format and bridges calls back to the MCP server.

Use Case Walkthroughs

1. Assign a GPT Agent to a Group

Setup:

Agent definition in DB:
  id: "gpt-creative"
  provider: "openai"
  model: "gpt-4o"
  personality: "You are a creative storytelling assistant. Be vivid and imaginative."
  containerImage: "cambot-agent-openai:latest"
  secretKeys: ["OPENAI_API_KEY"]

Group "story-club" updated:
  agentId: "gpt-creative"

Message flow:

  1. User sends @Andy tell us a story about dragons in the story-club group.
  2. Router matches trigger, enqueues message check.
  3. Orchestrator looks up story-club.agentId → "gpt-creative".
  4. Fetches AgentDefinition → gets containerImage: "cambot-agent-openai:latest", secretKeys: ["OPENAI_API_KEY"].
  5. readSecrets(["OPENAI_API_KEY"]) reads only OPENAI_API_KEY from .env.
  6. buildContainerArgs() uses "cambot-agent-openai:latest" image.
  7. Container starts, reads ContainerInput from stdin, calls OpenAI API.
  8. Response written to stdout with sentinel markers.
  9. Host parses ContainerOutput, routes to WhatsApp.

Everything outside step 7 is identical to a Claude run.

2. Council: 3 Agents Deliberate on a Problem

Setup: A new council-mode feature where the main agent delegates to multiple sub-agents:

Agent definitions:
  - id: "claude-analyst", provider: claude, personality: "Analytical and precise"
  - id: "gpt-creative", provider: openai, personality: "Creative and exploratory"
  - id: "grok-researcher", provider: grok, personality: "Data-driven, cites sources"

Flow:

  1. User asks: @Andy Should we invest in project X? Get multiple perspectives.
  2. The primary agent (Claude) receives the request via normal flow.
  3. Using the existing Agent Teams (TeamCreate/SendMessage) SDK feature, the primary agent spawns sub-queries — but these go through the host's container runner.
  4. Alternatively, the primary agent uses send_message MCP tool to post each sub-agent's analysis back to the group, and the host orchestrates parallel containers.

Implementation note: Council mode is a higher-level feature that builds on multi-provider. The foundation this design provides (per-agent container images + per-agent sessions) is sufficient. The council orchestration logic (how to merge perspectives, voting, etc.) is a separate concern.

3. Specialized Agents: Per-Task Routing

Setup:

Group "dev-team":
  Default agentId: "claude-coder"

Scheduled tasks:
  - "Daily code review" → agentId: "claude-coder" (best at code)
  - "Weekly newsletter draft" → agentId: "gpt-creative" (best at prose)
  - "Morning news briefing" → agentId: "grok-researcher" (real-time web)

Extension to scheduled tasks: The scheduled_tasks table gains an optional agent_id column. When the scheduler fires a task:

  1. If task.agent_id is set, use that agent definition.
  2. Otherwise, fall back to the group's default agentId.
  3. Session key is (group_folder, agent_id) — each agent maintains its own context.

Implementation Sequence

Phase 1: Foundation (No New Providers Yet)

  1. Add AgentDefinition type and agent_definitions DB table.
  2. Seed claude-default agent definition matching current behavior.
  3. Add agentId to RegisteredGroup (default: 'claude-default').
  4. Refactor container-runner.ts: parameterize readSecrets(), buildContainerArgs(), and buildVolumeMounts() to accept AgentDefinition.
  5. Remove global CONTAINER_IMAGE from config.
  6. All existing groups continue working identically — claude-default resolves to current image and keys.

Phase 2: Session Isolation

  1. Migrate sessions table to composite key (group_folder, agent_id).
  2. Update getSession()/setSession() to accept agentId.
  3. Existing sessions migrate with agent_id = 'claude-default'.

Phase 3: OpenAI Provider Container

  1. Create agent-runner-openai/ with OpenAI SDK agent runner.
  2. Implement the stdin/stdout contract with sentinel markers.
  3. Bridge MCP tools to OpenAI function-calling format.
  4. Build cambot-agent-openai:latest image.
  5. Add gpt-default agent definition.

Phase 4: Grok Provider Container

  1. Fork OpenAI runner with xAI base URL.
  2. Build cambot-agent-grok:latest image.
  3. Add grok-default agent definition.

Phase 5: Agent Management

  1. MCP tools for the main agent to manage agent definitions (CRUD).
  2. MCP tool to assign agents to groups.
  3. UI support in cambot-core-ui for viewing/editing agent assignments.

Phase 6: Council Mode

  1. Design council orchestration protocol.
  2. Implement parallel agent invocation for a single prompt.
  3. Response merging/voting strategies.

Risks and Mitigations

RiskMitigation
OpenAI/Grok lack Claude's tool-use depth (no Read, Write, Edit built-ins)Provider runners implement tool bridging. Start with Bash + MCP tools only. Expand tool support incrementally.
Session resume semantics differ across providersEach provider manages its own session format. The host only stores an opaque session ID.
Secret sprawl (many API keys in .env)secretKeys array ensures each container only receives what it needs. No over-sharing.
Increased container image countOne-time build cost. Images share a base layer. CI builds all images in parallel.
MCP tool bridging complexity for non-Claude providersStart with the 6 MCP tools (send_message, schedule_task, list_tasks, pause_task, resume_task, cancel_task). These are simple request/response — easy to bridge.

Open Questions

  1. Agent definition storage — DB table (as proposed) vs. config file (agents.json)? DB is more flexible for runtime management; config file is simpler for initial setup.
  2. Personality injection — Append to system prompt, or use provider-specific mechanisms (Claude's systemPrompt.append vs. OpenAI's system message)?
  3. Tool parity — Should non-Claude agents get the full tool suite (Read, Write, Edit, Glob, Grep), or start with Bash + MCP only? Full parity requires reimplementing each tool in the provider's function-calling format.
  4. Council orchestration — Host-driven (spawn N containers in parallel) vs. agent-driven (primary agent delegates via Agent Teams)? Host-driven is simpler but less flexible.
